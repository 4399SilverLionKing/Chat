# WeFlow 聊天数据访问

## 适用场景

- 用户问的是某个具体对象，而且答案应该建立在聊天记录上
- 用户想先看最近聊天，再决定怎么分析或怎么回
- 用户需要联系人信息、最近消息、已有画像或相关上下文

## 工作流

1. 先判断目标联系人是否已经明确
2. 如果不明确，先要求或解析 `wxid` / `wechat_id`
3. 优先通过仓库命令取数：
   - `show-contact`
   - `fetch-chat-context`
   - 必要时再用现有重分析命令
4. 联系人确认且拿到 `wxid` 后，先检查仓库内是否已有画像文件 `data/profiles/<wxid>.md`
5. 如果画像已存在，优先复用已有画像；只有画像不存在，或现有画像明显不足以支撑当前问题时，再考虑 `analyze-chat-profile` 等重分析命令
6. 只加载当前问题真正需要的上下文，不要一上来把所有历史都拉满

## 工具使用原则

- 获取聊天前先确认联系人
- 用 `show-contact` 做结构化联系人确认
- 用 `fetch-chat-context` 获取最近聊天或时间范围内聊天
- 联系人确认后，先检查 `data/profiles/<wxid>.md` 是否存在
- 只有确实需要长期画像，且 `data/profiles/<wxid>.md` 不存在或明显不足时，才调用 `analyze-chat-profile`
- 只有用户明确要重型回复建议产物时，才调用 `generate-reply-strategy`

## 数据优先级

优先使用“足够但不过量”的数据源，顺序如下：

1. 已确认的联系人身份
2. 仓库内已有画像文件 `data/profiles/<wxid>.md`
3. 最近聊天上下文
4. 已有 reply-strategy 结果
5. 更重的重新生成命令

## 失败处理

- 联系人无法确认时，要要求更明确的标识，不要猜
- 最近聊天不可用时，要明确说明接下来只能给通用建议
- 命令失败时，简短说明失败点，并选择下一个可用信息源
- 绝对不要虚构没有看到的聊天内容
